# Vigil v0.2.2 ‚Äî design

> Design rationale for v0.2.2. Produced by a dual-model consensus design process (two independent reasoning models ‚Äî GPT-5.5 high and Claude Opus 4.7 1M high ‚Äî investigated the same brief in parallel, cross-pollinated their findings, then a third blind-spot critique pass surfaced shared assumptions) before any code was written. Investigation artifacts live in the session workspace; this document is the synthesised, scoped, implemented result. Parent: [`docs/0.2.0-design.md`](0.2.0-design.md).

## What changes

v0.2.2 adds a **battery floor for both Lid-Awake and Caffeinate**: while running on battery power, the hold agent disables the feature once the battery percentage drops to a user-configured value. For Lid-Awake the agent also restores the saved `pmset` profile so the Mac can sleep normally instead of running flat. The shape closely follows the existing `Duration` / `pmset-snapshot` / `LaunchAgent` patterns established in v0.2.0 ‚Äî one optional field on `FeatureSession`, one CLI flag, one menu app control, one new `EndReason`, one new `DispatchSourceTimer` in `HoldEngine.run`. No new privilege surface, no IPC contract bump, no schema bump.

Feature-specific differences are intrinsic to the features, not artifacts of the design:

| Aspect | Lid-Awake | Caffeinate |
|--------|-----------|------------|
| Menu-app default | **on at 20%** (unattended ‚Üí safety) | **off** (user-present ‚Üí opt in) |
| Approved helper required | yes (pmset restore needs non-interactive privilege) | no (no pmset; just IOKit assertion release) |
| Trip teardown | two-phase (release IOKit, then bounded-retry pmset restore) | single-phase (release IOKit, remove sentinel, exit) |
| Notification copy | reason-aware AND restore-aware (lies-prevention) | reason-aware only |

The CLI flag (`--battery-floor <int>`), the persistence field (`FeatureSession.batteryFloorPercent`), the `EndReason.batteryThreshold` case, the menu app's `BatteryFloorControl` + `BatteryFloorMenu` views, the `HoldEngine.run` watchdog timer, and the AppStorage keys are all feature-agnostic ‚Äî the only branches are the teardown shape and the helper requirement.

## Operational semantics

The trip predicate is `isBatteryPower && batteryPercent <= floor`. Both conditions matter:

- **On battery only.** A Mac on AC at 18% (genuinely common during charging-from-low) must not trip. The `isBatteryPower` gate is what makes the feature do nothing while plugged in.
- **`<=`, not `<`.** Users intuitively mean "disable at 20%" ‚Äî the trip fires the moment the reading reaches the configured number, not after.

The trip is **one-way**: once it fires, the agent emits `StatsEvent.sessionEnded(.batteryThreshold)`, releases assertions, restores visuals, restores the `pmset` snapshot, removes session state, and exits. AC restored later does NOT auto-re-enable Lid-Awake. The user re-enables explicitly. Auto-re-enable on AC restore would require a pause/resume state machine inside the agent (keeping it alive across the disable while holding no assertions, then re-acquiring on AC notification, then handling the edge cases where Sparkle/shutdown/logout interleaves) ‚Äî that's a different feature shape entirely, deliberately deferred.

Hysteresis is irrelevant for a one-way exit. A single below-threshold sample fires the exit; subsequent samples are moot because the agent has already exited.

Optimized Battery Charging (Apple's ~80% hold while plugged in) is invisible to the trip ‚Äî OBC holds the displayed percent at 80% only while on AC, so the `isBatteryPower` gate is false. On unplug, the displayed percent immediately becomes the real percent and the predicate works as designed.

**Pre-arm refusal.** Enabling Lid-Awake on battery at-or-below the configured floor refuses up front with an explicit error rather than starting a session that lives for two seconds. The refusal is bypassed when the agent is being re-armed after a Sparkle update (`--rearm` flag) ‚Äî the agent's startup sample then fires a clean `.batteryThreshold` trip instead of orphaning the on-disk session record.

## Persistence

The threshold lives on `FeatureSession` as one optional field:

```swift
public let batteryFloorPercent: Int?
```

`Optional<Int>` makes the JSON gracefully default to `nil` for sessions persisted by 0.2.1 (the decoder ignores missing keys for optionals). No migration code, no schema bump.

The threshold **must** ride on `FeatureSession` and not in a sidecar file. The reason is `AppCoordinator.reArmAgentsIfNeeded`: it rebuilds the agent's enable command after a Sparkle relaunch from the persisted session. Anything stored in a sidecar (or, worse, in the menu app's `@AppStorage`) would be lost on the rearm path for CLI-originated sessions. Single source of truth ‚Üí single storage spot.

## Battery percentage source

The agent's enforcement loop and the CLI's pre-arm check both read `pmset -g batt` and call the new `BatteryState.internalBatteryPercentInt` helper, which strips the trailing `%` and clamps to `0...100`. We considered IOKit (`IOPSCopyPowerSourcesInfo` + `IOPSCopyPowerSourcesList`) ‚Äî it would give us typed `Int` and `Bool` values without subprocess forks or locale-sensitive string parsing ‚Äî but the patch-release ethos won the dual-model consensus: the pmset helper is ~5 lines of additive code while the IOKit version would add ~30 lines + a new framework import + careful CFTypeRef ownership. Both sources read the same SMC-backed registry, so there is no consistency benefit. We can revisit in 0.3.0 if profiling ever surfaces a cost (it won't, at 60-second polling).

## CLI surface

```
vigil lid-awake on [--duration <preset>] [--force-battery] [--battery-floor <int>] [--no-dim-display] [--no-dim-keyboard]
vigil lid-awake toggle [--duration <preset>] [--force-battery] [--battery-floor <int>]
```

`--battery-floor` accepts an integer in `1...99`. `0` and `100` are degenerate (never trip / always trip) and rejected loudly. Out-of-range and non-integer values are usage errors. The CLI requires `vigil approve-all` to have been run before accepting `--battery-floor`, because the agent's trip handler restores the privileged `pmset` profile non-interactively from inside the `gui/<uid>` LaunchAgent context.

`--battery-floor` and `--force-battery` are orthogonal. A user wanting "start on battery, exit at 20%" passes both. A user wanting "only ever run on AC, with a safety floor in case AC drops" passes only `--battery-floor` (the agent will trip the moment AC is unplugged below the floor).

`--rearm` is an internal flag used by the menu app's `reArmAgentsIfNeeded` to skip the at-or-below-floor pre-arm refusal during Sparkle relaunch. Hidden from `--help`.

## Agent enforcement

`HoldEngine.run` grows a fourth `DispatchSourceTimer` ‚Äî the **battery watchdog** ‚Äî gated on `feature == .lidAwake && session?.batteryFloorPercent != nil`. Three deliberate choices:

1. **Cadence matches the existing 60-second belt-and-suspenders.** A dev reading `HoldEngine.run` should not have to ask why timer #4 ticks at a different rate than timer #2. At realistic discharge rates (~1%/min idle with the lid closed, slightly more under load) trip latency past the floor is bounded at ~1%.
2. **Sampling runs on a background utility queue (`DispatchQueue(label: "com.vigil.app.battery-watchdog", qos: .utility)`), not `.main`.** `BatteryState.load` forks `/usr/bin/pmset`; under disk/CPU pressure that fork can stall for seconds. If it ran on `.main`, the 1Hz sentinel watchdog, the lid-poll timer, and the SIGTERM/SIGINT dispatch sources would all starve. The trip dispatch hops back to `.main` for state mutation.
3. **An immediate sample fires before installing the timer.** This handles the "armed below floor at rearm" path (Sparkle update on low battery ‚Üí fresh agent's first action is to trip cleanly) and the "armed on AC then immediately unplugged" path (sub-second trip instead of waiting up to 60s).

## Two-phase teardown

The existing `HoldEngine.handleExpiry` body was refactored into a shared private static helper `terminateLidAwake(reason:assertionIDs:)` with two distinct phases. Both the timer-expiry path (`handleExpiry(.timerExpired)`) and the battery-trip path (`handleBatteryTrip(.batteryThreshold)`) call it.

**Phase A ‚Äî always-safe (no privilege required).** Emit `StatsEvent.sessionEnded` while session + telemetry are still on disk (idempotent via the session-ID de-dup in `StatsLog`), release the IOKit assertions, restore visual state best-effort, remove lid telemetry. After Phase A the user-visible awake state is gone *except* for `pmset disablesleep=1`.

**Phase B ‚Äî privileged.** Attempt `restorePmsetSnapshot()`.

- On success: remove the sentinel, clear the session record, `exit(0)`.
- On failure: **keep the agent process alive**, keep sentinel + session on disk so `Status.build`'s `active = (sleepDisabled ?? false) && (agentRunning || sentinelExists)` honestly reflects the degraded state. Retry every 30 seconds with a hard cap of **2 attempts** (1 minute total). Log loudly to the agent's stderr file on each attempt.
- After cap: remove sentinel + session, then `exit(1)`. The sentinel removal MUST precede the exit; otherwise launchd's `KeepAlive.PathState` would re-spawn us in a loop.

The retry cap is intentionally short. A longer window (e.g. 10 attempts √ó 30s = 5 minutes) would invite a confused user to run `vigil lid-awake on` to "reset" the stuck Mac, double-bootstrapping the agent against an in-flight retry loop. 1 minute is enough for a transient sudo/helper hiccup to clear without inviting the foot-gun.

`handleSignaledExit` (SIGTERM/SIGINT path) does NOT fold into `terminateLidAwake` ‚Äî it intentionally skips the pmset restore (Sparkle bootout case, per the existing v0.2.0 design rationale: the brief gap between bootout and re-arm should not flap the lid-sleep behaviour). A small `terminating` guard with an `NSLock` prevents the retry loop and a concurrent signal handler from re-entering each other.

## Forward-compatible `EndReason` decoder

Adding a new case to `StatsEvent.EndReason` would normally be a quiet additive change. But the rubber-duck blind-spot pass surfaced a subtle hazard: `StatsLog.readAllUnsafe` uses `compactMap { try? decoder.decode(...) }`. A raw value the local binary doesn't recognise throws `DecodingError.dataCorrupted`, `try?` swallows it, and `compactMap` silently drops the entire `sessionEnded` event ‚Äî not just the field. The matching `sessionStarted` then becomes a permanent dangling session in `StatsAggregate.fold`.

So we add **two** new cases:

```swift
case batteryThreshold
case unknown
```

`.unknown` is a forward-compat sentinel. A custom `init(from:)` maps any unrecognised raw value to `.unknown` rather than throwing. Never emitted by this binary; only used on the read side. The fix lands in 0.2.2 so future enum additions in 0.2.3+ won't silently corrupt the historical log when read by an older 0.2.2 binary mid-Sparkle. (The 0.2.2 ‚Üí 0.2.1 rollback case can't be retroactively fixed because 0.2.1 is shipped, but downgrades are rare.)

## Menu app

### Battery-floor control

A new `BatteryFloorMenu` view modelled byte-for-byte on the existing `DurationMenu` ‚Äî same visual language users already know. Five presets (10/15/20/25/30%); coarser loses meaningful choice, finer is decision paralysis with no practical difference. The CLI keeps the full `1...99` range for power users who really want 17% or whatever.

The menu is paired with a `Toggle` ("Disable on low battery") and an always-visible caption ("Auto-disables Lid-Awake on battery so the Mac sleeps normally."). All three live in the Lid-Awake card accessory block, above `LidAwakeVisualToggles`. Disabled while the feature is active or while another action is in flight, and disabled on a no-battery Mac (`StatusReport.battery.percent == nil`) with an explanatory tooltip.

### Default state and disclosure

The toggle defaults to **on at 20%**. The user's stated motivation ‚Äî "let the Mac sleep normally instead of running entirely flat" ‚Äî argues for the safety net being active by default. But default-on in a patch release would be a silent behaviour change without disclosure.

Two disclosure surfaces address this:

1. The **always-visible caption** under the toggle describes what it does at every glance. Users who don't read it still get the safety net; users who do read it understand the behaviour.
2. The **battery-confirmation modal copy** (`AppCoordinator.turnOn`, the second-click-to-confirm flow) is now a function of `(onBattery, floorEnabled, floorPercent)`. When the user is on battery with the floor enabled at 20%, the modal says: *"On battery ‚Äî Lid-Awake will auto-disable below 20%. Click Turn On again to confirm."* When the floor is off, it says: *"On battery ‚Äî no battery floor configured. Lid-Awake will run until you stop it. Click Turn On again to confirm."* ‚Äî stronger wording when there's no safety net.

The combination makes default-on defensible without introducing a one-time toast (which would require a `hasSeen` flag and additional `@AppStorage` machinery).

### Reason-aware, restore-aware notifications

`ExpiryNotifier.notify` now reads `lastEndReason` and `sleepDisabled` from the status snapshot at the moment of the active‚Üíinactive transition:

- `.batteryThreshold` AND `sleepDisabled == false` ‚Üí *"Lid-Awake disabled ‚Äî battery low."*
- `.batteryThreshold` AND `sleepDisabled == true` ‚Üí *"Lid-Awake ended ‚Äî power settings could not be restored. Run `vigil doctor`."*
- `.timerExpired` / `.interrupted` ‚Üí *"Lid-Awake ended."* (existing copy)
- `.userDisabled` ‚Üí no notification (consistent with current behaviour ‚Äî the user already knows).

The `sleepDisabled` gate is what makes the safety feature honest. If the agent's pmset-restore retry cap exhausted, saying "battery low ‚Äî disabled cleanly" would lie about the safety feature working. The user needs to know they have a stuck-pmset state and how to recover it.

### Sparkle re-arm

`AppCoordinator.reArmAgentsIfNeeded` rebuilds the rearm command from the **persisted session's `batteryFloorPercent`**, not from the menu app's `@AppStorage`. The on-disk session is the truth for the currently-running session; `@AppStorage` is only the picker default for the *next* enable. This is what makes a CLI-originated session (one that never touched AppStorage) round-trip correctly through a Sparkle relaunch.

The rearm path also appends `--rearm` so the CLI's at-or-below-floor pre-arm refusal is skipped. The agent's immediate startup sample then fires a clean `.batteryThreshold` trip if needed, instead of leaving the session record orphaned on disk.

## Status JSON / human output

Two additive optional fields:

- `features[].session.batteryFloorPercent: Int?` ‚Äî surfaces automatically once the field is on `FeatureSession`.
- `features[].stats.lastEndReason: StatsEvent.EndReason?` ‚Äî plumbed through `Status.build` from `StatsAggregate.PerFeature.lastEndReason` (which has always existed but wasn't surfaced).

`schemaVersion` stays at 1 ‚Äî both additions are non-breaking on the JSON wire. The `EndReason` decoder fix above is what makes adding new enum cases also safe.

Human status now prints `  Battery floor: 20%` when the session has a floor and `  Last end reason: battery-threshold` (and the other cases) when stats are populated.

## Edge cases worth knowing

- **Battery percentage flap near the threshold.** Readings can oscillate by ¬±1% under load. Irrelevant for the one-way exit: even if 19‚Üí20‚Üí19, the first below-floor sample wins and subsequent samples don't matter because the agent has already exited. This is one of the strongest arguments for the one-way design.
- **Stale percentage immediately after wake from sleep.** Observed on Apple Silicon for ~1‚Äì3 seconds after wake. The 30-second poll cadence provides a natural soak; the first post-wake tick is at least 30s after wake, well after IOKit has settled. If we ever move to `IOPSNotificationCreateRunLoopSource` we'd need explicit debouncing; with polling we don't.
- **Threshold above current battery at enable time.** Pre-arm refusal in `LidAwakeController.enable` with an explicit error: *"Battery is at 18%, at or below the configured floor of 20%."* No silent immediate-self-disable in the agent; loud refusal in the foreground process is more comprehensible. The `--rearm` flag bypasses this so a Sparkle relaunch on now-low battery still results in a clean session-ended event (handled by the agent's startup sample).
- **Desktop Macs (no internal battery).** `BatteryState.internalBatteryPercentInt` returns `nil`; `isBatteryPower` is `false` (no battery to be on). The trip never fires by construction. The menu app's control is disabled with a tooltip explaining why.
- **Battery crosses the threshold during an interactive disable.** `StatsLog.append` is idempotent on `sessionID` ‚Äî the first `sessionEnded` for the session wins, the second is silently dropped. The pmset restore is also idempotent; IOKit assertion release is idempotent. Nothing to do.
- **Helper revoked mid-session.** The agent's retry loop catches this. After 2 √ó 30s attempts, the agent exits(1) with the sentinel removed (so launchd doesn't re-spawn) and surfaces the degraded state via `vigil status` / `vigil doctor`. The notification body honestly reports the failure.

## What was deliberately rejected

- **Symmetric default-on for Caffeinate.** Caffeinate is the user-present feature; auto-disable mid-presentation/video would be more surprising than helpful. The toggle exists, defaults off, and is one click away if a user wants it. Lid-Awake's case is different (unattended), which justifies default-on there.
- **Auto-re-enable when AC is restored.** Would require an in-process pause/resume state machine, AC-restore IOKit notifications, and a "while paused, hold no assertions but keep the agent alive" mode. Strictly a different feature; out of scope.
- **Hysteresis with a re-enable band.** Has no surface to act on once the trip is one-way; pure complexity, zero benefit.
- **IOKit `IOPSCopyPowerSourcesInfo` instead of `pmset -g batt`.** Cleaner long-term but ~30 LOC + framework + CFTypeRef ownership vs ~5 LOC for the pmset helper. Patch-release ethos wins. Both sources read the same registry.
- **`IOPSNotificationCreateRunLoopSource` instead of a 60s poll.** Cleaner architecturally ‚Äî zero polling, callback on every state change ‚Äî but introduces `CFRunLoopSource` lifetime management to a file that uses GCD exclusively, and the callback shape adds re-entry concerns during teardown. The 60s poll already meets the latency requirement.
- **A sidecar `battery-floor-options.json` file.** Would let CLI-originated sessions silently lose the threshold on Sparkle relaunch. Putting it on `FeatureSession` is the single-source-of-truth shape.
- **`StatusReport.currentSchemaVersion` bump.** Adding optional fields to existing nested structs is non-breaking. Only the unknown-enum-case hazard required a fix, and it was addressed via a tolerant decoder, not a schema version.
- **IPC contract version bump.** No sudoers verb / pmset allowlist / wire-format change ‚Äî the new battery-trip path routes through `Privilege.runApprovedHelperPMSetBatch` with byte-identical commands to today's `handleExpiry`.
- **A free-typed text field for the threshold.** Accepts nonsense, requires inline validation UI, breaks tab order. `BatteryFloorMenu` matches the existing `DurationMenu` precedent.
- **A top-level `batteryFloorTripped` flag in the status JSON.** Once the agent exits, the session file is gone; the existing `active ‚Üí inactive` transition + `lastEndReason` already conveys the outcome.
- **Removing `--force-battery` and gating battery enable purely on `--battery-floor`.** Would change CLI semantics (breaking 0.2.X compatibility) and conflate two different protections. `--force-battery` = "I consciously want to start on battery", `--battery-floor` = continuous safety during the session. Both have a place.
- **Onboarding screen mention.** Would block on copy + screenshots. The always-visible caption + release notes are sufficient.
- **A one-time `lastMessage` toast for first-after-upgrade disclosure.** The always-visible caption is simpler and doesn't require a `hasSeen` flag.
- **A 10-attempt √ó 30s retry window** (initial design). The 5-minute window invites confused-user `vigil lid-awake on` "resets" that double-bootstrap the agent. 2 √ó 30s (1 minute total) is the right trade-off.
