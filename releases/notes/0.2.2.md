## Lid-Awake battery floor

`vigil lid-awake` now supports a **battery floor**: while running on battery power, the agent disables Lid-Awake once the battery drops to a configured percentage, restoring the saved `pmset` profile so the Mac can sleep normally instead of running flat.

- New CLI flag: `vigil lid-awake on --battery-floor <int>` (range 1–99).
- New popover control: **Disable on low battery** toggle + percentage picker inside the Lid-Awake card.
- **Default in the menu app is on at 20%.** Existing 0.2.1 users updating via Sparkle get the safety net by default; the toggle is always visible in the card with an explanatory caption, and the on-battery enable modal discloses the configured floor before each Turn On.
- Trip is one-way: plugging AC back in does NOT auto-re-enable Lid-Awake.
- Requires `vigil approve-all` so the agent can restore power settings non-interactively at trip time.

If the privileged restore fails at trip time (e.g. approval was revoked mid-session), the agent retries twice on 30-second intervals, then exits. `vigil status` honestly reports the degraded state until you run `vigil lid-awake off` to restore power settings interactively, and the menu app's notification says `Lid-Awake ended — power settings could not be restored. Run \`vigil doctor\`.` instead of pretending the safety feature worked.

## Diagnostics

- `vigil status` (and the JSON status payload) now surface the configured battery floor on the active session, plus the most recent end reason (`user-disabled` / `timer-expired` / `interrupted` / `battery-threshold`) per feature.
- Reason-aware notification copy: `Lid-Awake disabled — battery low.` instead of the generic ending notification when the battery floor trips cleanly.

## Compatibility

- Additive only — no schema bump, no IPC contract bump, no breaking CLI / JSON changes.
- Sessions persisted by 0.2.1 decode unchanged.
- The `EndReason` decoder is now forward-compatible: future enum cases (0.2.3+) will land as `unknown` rather than silently dropping `sessionEnded` events from the stats log.
